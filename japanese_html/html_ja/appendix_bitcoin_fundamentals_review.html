付録A：ビットコインのファンダメンタルズレビュー

ライトニングネットワークは、複数のブロックチェーンの上で動作することが可能ですが、主にビットコインに軸足を置いています。ライトニング・ネットワークを理解するためには、ビットコインとその構成要素に関する基本的な理解が必要です。

ビットコインについてもっと学ぶために使える良い資料がたくさんあります。その中には、オープンソースライセンスのもとGitHubで見つけることができる、Andreas M. Antonopoulosによる関連書籍『Mastering Bitcoin, 2nd Edition』も含まれます。ただし、この本の準備のために他の本を全部読む必要はありません

この章では、ビットコインについて知っておくべき最も重要な概念を集め、ライトニング・ネットワークの文脈で説明しました。こうすることで、ライトニングネットワークを把握するために必要なことを、雑念を入れずに正確に学ぶことができます。

キーとデジタルシグネチャ

ビットコインは、コンピュータのセキュリティに広く利用されている数学の一分野である暗号技術に基づいていることをご存知でしょうか。暗号学は、秘密を明かすことなくその秘密を知っていることを証明したり（デジタル署名）、データの真正性を証明したり（デジタル指紋）するためにも使用されることがあります。このような暗号学的証明は、ビットコインにとって重要な数学的ツールであり、ビットコインのアプリケーションで広く使用されています。

ビットコインの所有権は、デジタルキー、ビットコインアドレス、およびデジタル署名によって確立されます。デジタルキーは実際にはネットワークに保存されていませんが、代わりにウォレットと呼ばれるファイル、または単純なデータベースにユーザーによって作成、保存されます。ユーザーのウォレット内のデジタルキーは、ビットコインプロトコルから完全に独立しており、ブロックチェーンを参照したり、インターネットにアクセスすることなく、ユーザーのウォレットソフトウェアによって生成および管理することができます。

ほとんどのビットコイン取引では、ブロックチェーンに含まれる有効なデジタル署名が必要です。これは秘密鍵でしか生成できないため、その鍵のコピーを持つ誰もがビットコインを管理することができます。 資金を使うために使われるデジタル署名は、暗号学で使われる用語である「証人」とも呼ばれます。ビットコイン取引における証人のデータは、使用される資金の真の所有権を証言するものです。鍵は、秘密鍵と公開鍵の2つで構成されています。公開鍵は銀行の口座番号のようなもので、秘密鍵は秘密の暗証番号のようなものだと考えてください。

秘密鍵と公開鍵

秘密鍵は、ランダムに選ばれた単なる数字です。実際には、多くの鍵を簡単に管理するために、ほとんどのビットコインウォレットは、決定論的導出アルゴリズムを使って、単一のランダムシードから一連の秘密鍵を生成しています。簡単に言えば、1つの乱数を使って、秘密鍵として使用される一見乱数のように見えるシーケンスを繰り返し生成しているのです。これにより、ユーザーはシードをバックアップするだけで、そのシードから必要なすべての鍵を導出することができるようになります。

Bitcoinは、他の多くの暗号通貨やブロックチェーンと同様に、楕円曲線をセキュリティに使用しています。ビットコインでは、楕円曲線secp256k1上の楕円曲線乗算が一方向性関数として使用されています。簡単に言えば、楕円曲線数学の性質上、ある点のスカラー倍算の計算は簡単だが、その逆（除算、離散対数）の計算は不可能なのである。

秘密鍵にはそれぞれ対応する公開鍵があり、秘密鍵から楕円曲線上のスカラー倍算で計算されます。簡単に言うと、秘密鍵kがあれば、それに定数Gを掛けて公開鍵Kを生成することができるのです。

  • K=k*G

この計算を逆算することは不可能である。公開鍵Kがあれば、秘密鍵kは計算できない。Gによる除算は楕円曲線数学では不可能である。その代わりに、ブルートフォースアタックと呼ばれる網羅的な処理で、kの可能な値をすべて試さなければならない。kは256ビットなので、古典的なコンピュータで可能な限りの値を試すと、この宇宙では利用できないほどの時間とエネルギーが必要になる。

ハッシュ

ビットコインやライトニングネットワークで広く使われているもう一つの重要なツールは、暗号ハッシュ関数で、特にSHA-256ハッシュ関数が有名です。

ハッシュ関数はダイジェスト関数とも呼ばれ、任意の長さのデータを受け取り、ハッシュ、ダイジェスト、またはフィンガープリントと呼ばれる固定長の結果に変換する関数です（SHA-256暗号ハッシュアルゴリズムを参照）。重要なのは、ハッシュ関数は一方向性関数であり、逆にしてフィンガープリントから入力データを計算することはできないことです。

図1.SHA-256暗号ハッシュアルゴリズム

例えば、コマンドライン・ターミナルを使って「Mastering the Lightning Network」というテキストをSHA-256関数に投入すると、次のような指紋が生成される。

$ echo -n "Mastering Lightning Network" | shasum -a 256

ce86e4cd423d80d054b387aca23c02f5fc53b14be4f8d3ef14c089422b2235de  -
チップ

ハッシュを計算するための入力は、プリ画像とも呼ばれる。

もちろん、入力の長さはもっと大きくてもいい。同じことを、サトシ・ナカモトのビットコイン白書のPDFファイルでも試してみよう。

$ wget http://bitcoin.org/bitcoin.pdf
$ cat bitcoin.pdf | shasum -a 256
b1674191a88ec5cdd733e4240a81803105dc412d6c6708d53ab94fc248f4f553  -

一文よりも長い時間をかけて、SHA-256関数は9ページのPDFを処理し、256ビットの指紋に「ダイジェスト」します。

さて、この時点で、サイズが無制限のデータを消化する関数が、固定サイズの数値であるユニークな指紋を生成することが可能なのか疑問に思うかもしれません。

理論的には、可能なプリ画像（入力）は無限にあり、フィンガープリントは有限であるため、同じ256ビットのフィンガープリントを生成するプリ画像は多数存在するはずです。2つのプリイメージが同じハッシュを生成する場合、これは衝突として知られています。

実際には、256ビットの数値は非常に大きいので、意図的に衝突を見つけることはできない。暗号ハッシュ関数は、衝突を探すのはエネルギーと時間のかかるブルートフォースな作業であり、現実的には不可能であることを前提に動作します。

暗号ハッシュ関数は、いくつかの有用な特徴を持つため、様々なアプリケーションで広く使用されている。それらは

決定論的

同じ入力では常に同じハッシュが生成されます。

不可逆

ハッシュの前像を計算することはできない。

耐コリジョン性

同じハッシュを持つ2つのメッセージを見つけることは、計算上不可能である。

非相関

入力が少し変わると出力が大きく変わるので、出力は入力と相関がないように見える。

一様/ランダム

暗号化ハッシュ関数は、256ビットの出力可能空間全体に一様に分布するハッシュを生成する。ハッシュの出力はランダムであるように見えるが、本当にランダムではない。

このような暗号ハッシュの特徴を利用すると、興味深いアプリケーションを構築することができる。

フィンガープリント

ハッシュは、ファイルやメッセージを一意に識別するためのフィンガープリントとして使用することができます。ハッシュは、あらゆるデータセットの普遍的な識別子として使用することができます。

インテグリティ・プルーフ

ファイルやメッセージのフィンガープリントは、フィンガープリントを変更しないとファイルやメッセージの改ざんができないため、その完全性を証明するものです。これは、ソフトウェアをコンピュータにインストールする前に、ソフトウェアが改ざんされていないことを確認するためによく使用されます。

コミットメント/否認防止

特定のプリ画像（数字やメッセージなど）のハッシュを公開することで、それを明かすことなくコミットすることができます。後でその秘密を公開すれば、公開されたハッシュが生成されるため、誰もがそれが先にコミットしたものと同じものであることを確認できます。

プルーフ・オブ・ワーク/ハッシュ研磨

ハッシュを利用して、前画像を繰り返し推測することによってのみ生成できる非ランダムなパターンをハッシュに示すことで、自分が計算を行ったことを証明することができる。例えば、ビットコインのブロックヘッダのハッシュは、多くのゼロビットで始まります。これを作り出す唯一の方法は、ヘッダーの一部を変更し、偶然にそのパターンが生成されるまで何兆回となくハッシュ化することです。

アトミシティ

リンクされた複数の取引で資金を使うための前提条件として、秘密のプリイメージを作ることができる。当事者のうち誰かがある取引を使うためにその前像を明かすと、他の当事者もその取引を使うことができるようになる。すべて、あるいはどれも支出可能になり、複数の取引にまたがる原子性が達成される。

デジタル署名

秘密鍵は、取引で使用される資金の所有権を証明することで、ビットコインを使用する際に必要となる署名を作成するために使用されます。

電子署名は、特定のメッセージに秘密鍵を適用して算出される数値である。

メッセージmと秘密鍵kが与えられたとき、署名関数Fsignは署名Sを生成することができる。

\[$ S = F_{sign}(m, k) $].

この署名Sは、公開鍵K（秘密鍵kに対応）とメッセージを持っている人なら誰でも独立に検証することができる。

\[$ F_{verify}(m, K, S) $].

Fverifyが真の結果を返した場合、検証者はメッセージmが秘密鍵kにアクセスできた誰かによって署名されたことを確認できる。重要なのは、デジタル署名は署名の時点で秘密鍵kを所有していることを証明し、kを明らかにしないことである。

デジタル署名は、暗号化ハッシュアルゴリズムを使用する。署名はメッセージのハッシュに適用され、メッセージmは指紋として機能する固定長のハッシュH(m)に「要約」される。

トランザクションのハッシュに電子署名を適用することで、署名は認可を証明するだけでなく、トランザクションデータを「ロック」し、その完全性を保証します。署名されたトランザクションは、変更されると異なるハッシュとなり、署名が無効となるため、変更することはできません。

署名の種類

署名は常に取引全体に適用されるとは限らない。署名の柔軟性を提供するため、ビットコインのデジタル署名には署名ハッシュタイプと呼ばれるプレフィックスが含まれており、これはトランザクションデータのどの部分がハッシュに含まれるかを指定するものである。これにより、署名はトランザクションのデータのすべて、または一部のみをコミットまたは「ロック」することができます。最も一般的な署名のハッシュタイプはSIGHASH_ALLで、署名されたハッシュの中にすべてのトランザクションデータを含めることでトランザクションのすべてをロックします。それに比べて、SIGHASH_SINGLE はトランザクションのすべての入力をロックするが、出力は1つだけである（入力と出力については次のセクションで詳しく説明する）。異なる署名のハッシュタイプを組み合わせることで、署名によってロックされるトランザクションデータの6つの異なる「パターン」を作り出すことができる。

署名のハッシュタイプについての詳細は、『Mastering Bitcoin, Second Edition』の第6章「Signature Hash Types」のセクションに記載されています。

ビットコイン取引

取引は、ビットコインシステムの参加者間の価値の移転をコード化するデータ構造です。

入力と出力

ビットコイン取引の基本的な構成要素は、取引出力です。取引出力はビットコイン通貨の不可分の塊であり、ブロックチェーンに記録され、ネットワーク全体によって有効であると認識されています。取引はインプットを消費し、アウトプットを作成する。トランザクションのインプットは、以前に記録されたトランザクションのアウトプットへの単なる参照である。このように、各トランザクションは以前のトランザクションのアウトプットを消費し、新しいアウトプットを作成する（トランザクションはインプットからアウトプットへ価値を転送するを参照）。

図2.取引はインプットからアウトプットへ価値を伝達する

ビットコインのフルノードは、未使用のトランザクション出力（UTXO）として知られる、利用可能で使用可能なすべての出力を追跡します。すべてのUTXOのコレクションはUTXOセットとして知られており、現在、数百万UTXOの数があります。UTXOセットは、新しいUTXOが作成されると増加し、UTXOが消費されると減少する。すべてのトランザクションは、トランザクションの入力として1つ以上のUTXOを消費し、トランザクションの出力として1つ以上のUTXOを生成することにより、UTXOセットの変化（状態遷移）を表す。

例えば、ユーザーであるアリスが10万サトシのUTXOを持っていて、それを使うことができると仮定しよう。アリスはBobに10万サトシを支払うために、1つの入力（既存の10万サトシの入力を消費する）と1つの出力を持つトランザクションを構築することができる。これでBobは10万サトシのUTXOを持ち、それを使うことができる。この新しいUTXOを消費して別のUTXOに使い、別のユーザーへの支払いとする新しい取引を作成し、これを繰り返す（Alice pays 100,000 satoshis to Bobを参照）。

図3.アリスがボブに100,000サトシを支払う。

取引出力は、サトシ単位で任意の（整数）値を持つことができる。ドルを小数点以下2桁まで分割してセントとするように、ビットコインは小数点以下8桁まで分割してサトシとすることができる。アウトプットは任意の値を持つことができるが、一度作ったものは割り切れない。これは、アウトプットの重要な特性であり、強調されるべきことである。アウトプットは、整数のサトシで表される、不連続かつ不可分の価値単位なのである。未使用のアウトプットは、取引によってのみその全体が消費される。

では、アリスがボブに5万サトシを支払いたいのに、不可分の10万サトシのUTXOしか持っていないとしたらどうでしょう？アリスは10万サトシのUTXOを消費（入力）し、2つの出力を持つトランザクションを作成する必要があります。1つはボブに5万サトシを支払い、もう1つはアリスに5万サトシを「お釣り」として返します（アリスがボブに5万サトルを支払い、自分には5万サトルをお釣りとして支払うを参照）。

図4.アリスはボブに50kサットを、自分に50kサットをお釣りとして支払う
チップ

変更出力について特別なことは何もなく、他の出力と区別する方法もない。最後の出力である必要はありません。複数の変更出力がある可能性もあるし、変更出力がない可能性もある。どの出力が他人に対するもので、どの出力が自分の所有するアドレスに対するもので、したがって "変更 "であるかは、トランザクションの作成者だけが知っています。

同様に、アリスがボブに85,000サトシを支払いたいが、50,000サトシのUTXOが2つある場合、彼女は2つの入力（彼女の50,000サトシUTXOを両方消費する）と2つの出力、つまりボブに85,000サトシを支払い、自分自身にお釣りとして15,000サトシを送る取引を作成しなければなりません（アリスが2つの50k入力を使ってボブへの85kサトシ支払いと自分へのお釣りとしての15kサトシ支払いと参照して下さい）。

図5.アリスは2つの50k入力を使って、ボブに85kサットを、自分に15kサットをお釣りとして支払う

前述の図と例は、ビットコイン取引が1つ以上の入力を結合（消費）し、1つ以上の出力を作成する方法を示しています。トランザクションは数百、数千のインプットとアウトプットを持つことができる。

チップ

ライトニング・ネットワークによって生み出された取引は複数のアウトプットを持ちますが、チャネルの利用可能残高全体が2つのチャネルパートナーの間で分割されるため、それ自体には「変化」がありません。

トランザクション・チェーン

すべてのアウトプットは、後続のトランザクションのインプットとして使用することができる。例えば、BobがChanに支払う取引で10,000 satoshiを使うことを決め、ChanがDinaに支払うために4,000 satoshiを使った場合、Alice pays Bob who pays Chan who pays Dinaのように展開されることになる。

ブロックチェーンに記録された他の取引で入力として参照された場合、出力は使われたとみなされます。アウトプットは、記録された取引がそれを参照していない場合、未使用（および支出可能）と見なされる。

インプットを持たない取引は、coinbase取引と呼ばれるビットコインのマイナーによって作られる特別な取引だけです。coinbase取引は、採掘から新しいビットコインを作成するため、アウトプットのみを持ち、インプットを持たない。他の取引はすべて、以前に記録された1つ以上のアウトプットをインプットとして使用する。

取引は連鎖しているので、ランダムに取引を選ぶと、その入力のどれかを、それを作った前の取引までさかのぼって追うことができる。それを繰り返していくと、最終的にはビットコインが最初に採掘されたcoinbaseのトランザクションにたどり着く。

図6.アリスがボブに支払い、ボブはチャンに支払い、チャンはディーナに支払う。

TxID：トランザクション識別子

ビットコインシステムにおける全ての取引は、取引IDまたは略してTxIDと呼ばれる一意の識別子（BIP-0030の存在を仮定）により識別されます。一意な識別子を生成するために、SHA-256暗号ハッシュ関数を使用してトランザクションのデータのハッシュを生成します。この「フィンガープリント」は、普遍的な識別子として機能する。取引はその取引IDによって参照することができ、一度取引がビットコインのブロックチェーンに記録されると、ビットコインネットワークのすべてのノードがこの取引が有効であることを知ることができる。

例えば、トランザクションIDは次のようなものです。

トランザクションデータのハッシュ化により生成されるトランザクションID
e31e4e214c3f436937c74b8663b3ca58f7ad5b3fce7783eb84fd9a5ee5b9a54c

これは、ビットコインのブロックチェーン上で見つけることができる実際の取引（「Mastering Bitcoin」の本の例として作成された）である。ブロックエクスプローラーにこのTxIDを入力して検索してみてください。

  • https://blockstream.info/tx/e31e4e214c3f436937c74b8663b3ca58f7ad5b3fce7783eb84fd9a5ee5b9a54c

またはショートリンク（大文字・小文字を区別します）を使用します。

アウトポイント出力識別子

すべてのトランザクションは一意のIDを持っているので、TxIDと出力インデックス番号を参照することで、そのトランザクション内のトランザクション出力を一意に特定することもできる。トランザクション内の最初の出力は出力インデックス0、2番目の出力は出力インデックス1、といった具合になる。出力識別子は一般にアウトポイント（outpoint）と呼ばれる。

慣習上、アウトポイントは、TxID、コロン、出力インデックス番号で記述します。

A outpoint：TxIDとインデックス番号で出力を識別する。
7957a35fe64f80d234d76d83a2a8f1a0d8149a41d81de548f0a65a8a999f6f18:0

出力識別子（outpoints）は、トランザクションを連鎖的に結びつける仕組みである。すべてのトランザクションの入力は、前のトランザクションの特定の出力への参照である。その参照はアウトポイントであり、TxIDと出力インデックス番号である。したがって、トランザクションは、アウトポイントを参照することで、それ自体が使用できる新しい出力を作成するために、特定のトランザクション（TxIDによる）から特定の出力（インデックス番号による）を「使用」する。

トランザクションの入力はアウトポイントを参照し、鎖を形成してアリス→ボブ→チャン→ディナというトランザクションの連鎖を提示し、今回はそれぞれの入力にアウトポイントを持つようにした。

図 7.トランザクションの入力は、チェーンを形成するアウトポイントを指す

Bobのトランザクションの入力は、Aliceのトランザクションを(TxIDで)参照し、 0のインデックスを持つ出力を参照する。

Chanのトランザクションの入力は、BobのトランザクションのTxIDとインデックスされた最初の出力を参照する。なぜなら、Chanへの支払いは出力#1だからである。BobのChanへの支払いでは、Bobのおつりは出力#0である[1]。

さて、アリスからボブへの支払いを見てみると、アリスはIDが6a5f1b3[...]のトランザクションの3番目（出力インデックス#2）の出力であるアウトポイントを使用していることがわかります。図にはその参照されたトランザクションはありませんが、アウトポイントからこれらの詳細を推論することができます。

ビットコインスクリプト

理解を完了するために必要なビットコインの最後の要素は、アウトポイントへのアクセスを制御するスクリプト言語である。ここまでは、"アリスがボブに支払う取引に署名する "と言うことで説明を簡略化してきました。しかしその裏には、より複雑な支出条件を実装するための複雑さが隠されているのです。最も単純で一般的な支出条件は、"次の公開鍵に一致する署名を提示する "です。このような支出条件は、ビットコインスクリプトというスクリプト言語で書かれたロッキングスクリプトとして各出力に記録されます。

Bitcoin Scriptは、極めてシンプルなスタックベースのスクリプト言語です。ループや再帰を含まないため、チューリング不完全です（任意の複雑さを表現できず、実行が予測可能であることを意味します）。プログラミング言語FORTH（現在は古くなっている）に馴染みのある方は、その構文やスタイルが分かると思います。

ビットコインスクリプトの実行

簡単に言うと、ビットコインのシステムは、ビットコインスクリプトをスタック上で実行して評価し、最終結果が「真」であれば、支出条件が満たされ、取引は有効であると見なします。

ビットコインスクリプトの非常に単純な例として、数字の2と3を足して、その結果を数字の5と比較する例を見てみましょう。

スクリプトのロックとアンロック

Bitcoin Scriptは、2つの部分から構成されています。

スクリプトのロック

これらはトランザクションのアウトプットに埋め込まれ、そのアウトプットを使うために満たさなければならない条件を設定します。例えば、アリスの財布はボブに支払う出力にロックスクリプトを追加し、それを使うにはボブの署名が必要であるという条件を設定します。

スクリプトのアンロック

これらはトランザクション入力に埋め込まれ、参照される出力のロックスクリプトで設定された条件を満たす。例えば、Bobは、デジタル署名を含むロック解除スクリプトを提供することで、先行する出力のロックを解除することができる。

簡略化したモデルを用いると、検証のために、アンロックスクリプトとロッキングスクリプトを連結して実行する（P2SHとSegWitは例外）。例えば、誰かがトランザクション出力をロックスクリプト「3 ADD 5 EQUAL」でロックした場合、それをトランザクション入力のロック解除スクリプト「2」で過ごすことができる。そのトランザクションを検証する人は、ロック解除スクリプト（2）とロックスクリプト（3 ADD 5 EQUAL）を連結して、その結果をビットコインスクリプト実行エンジンで実行する。その結果、TRUEが得られ、私たちはその出力を使うことができる。

明らかに、この単純化された例は、実際のビットコインの出力をロックするには非常に不適切な選択です。誰でも「2」という答えを提供することで出力を使用することができます。したがって、ほとんどのロックスクリプトは、秘密の知識を示す必要があります。

公開鍵へのロック(Signature)

ロックスクリプトの最も単純な形式は、署名を必要とするものである。アリスがボブに50,000サトシを支払うトランザクションを考えてみましょう。アリスがボブに支払うために作る出力には、ボブの署名を必要とするロックスクリプトがあり、次のようになります。

ボブの秘密鍵によるデジタル署名を必要とするロックスクリプト
<Bob 公開鍵> CHECKSIG

CHECKSIG演算子は、スタックから署名と公開鍵の2つの項目を取り出します。見ての通り、Bobの公開鍵はロッキングスクリプトにあるので、足りないのはその公開鍵に対応する署名である。なぜなら、公開鍵に一致するデジタル署名を生成するのに必要な対応する秘密鍵は、ボブしか持っていないからです。

このロックスクリプトを解除するために、ボブは自分のデジタル署名のみを含むロック解除スクリプトを提供する。

ボブの秘密鍵のデジタル署名を含む（のみ）ロック解除スクリプト
<ボブ・サイン>

ロックスクリプト（出力）とアンロックスクリプト（入力）を示すトランザクションチェーンでは、アリスのトランザクションのロックスクリプト（ボブに支払う出力）とボブのトランザクションのアンロックスクリプト（その出力を使用する入力）が見える。

図9．ロッキングスクリプト（出力）とアンロッキングスクリプト（入力）を示すトランザクションチェーン

ボブの取引を検証するために、ビットコインノードは次のようなことをする。

  1. 入力（<Bob Signature>）からロック解除スクリプトを抽出します。

  2. 使おうとしているアウトポイント(a643e37...3213:0)を調べてみてください。これはアリスの取引であり、ブロックチェーン上で見つけることができます。

  3. そのアウトポイントからロックスクリプトを抽出します（<Bob PubKey> CHECKSIG）。

  4. ロック解除スクリプトをロックスクリプトの前に置いて、1つのスクリプトに連結する（<Bob Signature> <Bob PubKey> CHECKSIG）。

  5. このスクリプトをBitcoin Script実行エンジンで実行し、どのような結果が得られるかを確認します。

  6. 結果が「真」であれば、Bobのトランザクションは、そのアウトポイントを使うための支出条件を満たすことができたので、有効であると推論する。

ハッシュ（秘密）へのロック

Lightning Networkで使用されているロックスクリプトのもう一つのタイプは、ハッシュロックである。これを解除するには、ハッシュに対する秘密のプリ画像を知っている必要があります。

これを実証するために、ボブに乱数Rを生成してもらい、それを秘密にしておくことにする。

R = 1833462189

ここで、ボブはこの数値のSHA-256ハッシュを計算する。

h = sha256(r)=>となります。
H = SHA256(1833462189) => です。
H = 0ffd8bea4abdb0deafd6f2a8ad7941c13256a19248a7b0612407379e1460036a

さて、ボブは前に計算したハッシュHをアリスに渡しますが、 数値Rは秘密にします。暗号ハッシュの特性上、アリスはハッシュの計算を「逆引き」して数Rを推測することができないことを思い出してください。

アリスはロックスクリプトで50,000サトシを支払うアウトプットを作成する。

ハッシュ256イコール

ここで、Hはボブがアリスに渡した実際のハッシュ値(0ffd8...036a)である。

このスクリプトを説明しよう。

HASH256 演算子は、スタックから値をポップし、その値の SHA-256 ハッシュを計算します。そして、その結果をスタックにプッシュします。

H の値がスタックにプッシュされ、次に EQUAL 演算子が 2 つの値が同じかどうかをチェックし、それに応じて TRUE または FALSE をスタックにプッシュします。

したがって、このロックスクリプトは、Rを含むロック解除スクリプトと組み合わせないと動作しないので、連結すると次のようになります。

r hash256 h イコール

ボブだけがRを知っているので、ボブだけが秘密の値Rを明らかにするロック解除スクリプトのトランザクションを生成することができます。

興味深いことに、ボブはRの値を他の人に渡すことができ、その人はそのビットコインを使うことができます。これによって、秘密の値Rはほとんどビットコインの「引換券」のようになり、それを持っている人は誰でもアリスが作った出力を使うことができるのです。これがどのようにライトニングネットワークの有用な特性であるかは、これから見ていくことにしましょう

マルチシグネチャースクリプト

ビットコインスクリプト言語は、エスクローサービスや複数のステークホルダー間の複雑な所有権設定を構築するために使用できる、マルチシグネチャービルディングブロック（プリミティブ）を提供します。ビットコインを使用するために複数の署名を必要とする取り決めは、マルチシグネチャスキームと呼ばれ、さらにK-of-Nスキームと規定されています。

  • Nは、マルチシグネチャ方式で識別される署名者の総数であり

  • Kは 定足数または閾値：支出を承認するための最小限の署名数です。

K-of-Nマルチシグネチャのスクリプトは以下の通りです。

K <PubKey1> <PubKey2> ... <PubKeyN> N CHECKMULTISIG

ここで、Nはリストアップされた公開鍵の総数（公開鍵1〜公開鍵N）、Kは出力を消費するために必要な署名の閾値である。

ライトニング・ネットワークは、2-of-2マルチシグネチャ方式を使用して決済チャネルを構築します。例えば、アリスとボブの間の支払いチャネルは、次のような2-of-2マルチシグネチャで構築されます。

2 <PubKey Alice> <PubKey Bob> 2 CHECKMULTISIG

前述のロックスクリプトは、次の署名の組を含むロック解除スクリプトで満たすことができる。

0 ＜シグ・アリス＞＜シグ・ボブ＞の場合

この2つのスクリプトを合わせると、複合バリデーションスクリプトとなる。

0 <Sig Alice> <Sig Bob> 2 <PubKey Alice> <PubKey Bob> 2 CHECKMULTISIG

マルチシグネチャのロックスクリプトは、ロックスクリプトのハッシュをエンコードしたビットコインアドレスで表すことができる。例えば、ライトニング決済チャンネルの最初の資金取引は、2つのチャンネルパートナーの2-of-2マルチシグのロックスクリプトをエンコードしたアドレスに支払われる取引です。

タイムロックスクリプト

ビットコインに存在し、ライトニングネットワークで広く使用されているもう一つの重要なビルディングブロックはタイムロックです。タイムロックとは、一定時間またはブロックの高さが経過しないと利用できない、利用制限のことです。これは、銀行口座から振り出された日付指定の小切手が、その小切手に書かれた日付より前に現金化できないのと同じようなものである。

ビットコインには、トランザクションレベルのタイムロックとアウトプットレベルのタイムロックという2つのレベルのタイムロックがあります。

トランザクションレベルのタイムロックは、トランザクションのnLockTimeフィールドに記録され、タイムロックが経過する前にトランザクション全体が受理されるのを防ぐことができます。トランザクションレベルのタイムロックは、今日ビットコインで最もよく使われているタイムロックの仕組みである。

出力レベルのタイムロックは、スクリプトオペレータによって作成されます。出力タイムロックには、絶対タイムロックと相対タイムロックの2種類があります。

出力レベルの絶対タイムロックはCHECKLOCKTIMEVERIFY演算子で実装され、会話ではCLTVと略されることが多い。絶対タイムロックは、絶対タイムスタンプやブロックハイトで時間制約を実装し、"not spendable before block 800,000" と同等の表現になる。

出力レベルの相対タイムロックはCHECKSEQUENCEVERIFYという演算子で実装されており、会話ではしばしばCSVと短縮されます。相対タイムロックは、トランザクションの確認に対して相対的な支出制約を実装し、"確認後1024ブロックまで支出できない "と同等に表現する。

複数の条件を持つスクリプト

Bitcoin Scriptの強力な機能の1つに、条件節としても知られるフロー制御があります。IF...THEN...ELSE という構成を使用するさまざまなプログラミング言語でのフロー制御は、おそらく皆さんもよくご存じでしょう。ビットコインの条件節は少し違って見えますが、本質的には同じ構成です。

基本的なレベルでは、ビットコイン条件付きオペコードを使用すると、論理条件の評価結果が「真」または「偽」の場合に、2通りのロック解除方法を持つロックスクリプトを構築できます。例えば、xがTRUEであれば、ロックスクリプトはAである ELSEであれば、ロックスクリプトはBである。

さらに、ビットコインの条件式は無限に入れ子にすることができます。つまり、ある条件節の中に別の条件節を入れ、その中に別の条件節を入れる、といったことが可能なのです。ビットコインスクリプトフローコントロールは、何百、何千もの可能な実行経路を持つ非常に複雑なスクリプトを構築するために使用することができます。ネストに制限はありませんが、コンセンサスルールにより、スクリプトの最大サイズ（バイト単位）に制限が課されます。

Bitcoin は IF、ELSE、ENDIF、NOTIF オペコードを使用してフロー制御を実装しています。さらに、条件式はBOOLAND、BOOLOR、NOTなどのブーリアン演算子を含むことができます。

一見すると、Bitcoinのフロー制御スクリプトは分かりにくいと感じるかもしれません。それは、ビットコインスクリプトがスタック言語であるためです。Bitcoin Scriptで「1 + 1」という算術演算を「1 1 ADD」と表現すると「後ろ向き」に見えるのと同じように、Bitcoinのフロー制御の節も「後ろ向き」に見えます。 Bitcoinのフロー制御句も "後ろ向き "に見える。

従来の（手続き型）プログラミング言語の多くでは、フロー制御は次のようなものである。

ほとんどのプログラミング言語におけるフロー制御の疑似コード
if (条件):
  条件が真のときに実行されるコード
else:
  条件が偽のときに実行されるコード
どちらの場合でも実行されるコード

Bitcoin Scriptのようなスタックベースの言語では、論理条件がIFの前に来るので、次のように「後ろ向き」になります。

ビットコインスクリプトのフロー制御
条件
IF
  条件が真のときに実行されるコード
ELSE
  条件が偽のときに実行されるコード
ENDIF
どちらの場合でも実行されるコード

Bitcoin Scriptを読むときは、評価される条件がIFオペコードの前に来ることを覚えておいてください。

スクリプトでフロー制御を行う

ビットコインスクリプトにおけるフロー制御の非常に一般的な使用方法は、複数の実行パスを提供し、それぞれがUTXOを換金する異なる方法を提供するロックスクリプトを構築することです。

簡単な例として、アリスとボブの二人の署名者がいて、どちらかが換金をすることができる場合を見てみましょう。マルチシグでは、これは1-of-2マルチシグスクリプトとして表現される。デモンストレーションのために、同じことをIF句で行うことにします。

IF
 <アリスのパブキー> CHECKSIG
ELSE
 <ボブのパブキー> CHECKSIG
ENDIF

このロックスクリプトを見て、あなたは不思議に思うかもしれません。「条件はどこにあるのだろう？IF句の前には何もないじゃないか！" と思うかもしれない。

この条件はロックスクリプトの一部ではない。その代わり、条件はロック解除スクリプトで提供され、アリスとボブがどの実行経路を選ぶかを「選択」できるようにする。

アリスはこれをロック解除のスクリプトで挽回します。

<アリスのシグ> 1

最後の1は、IF節がアリスが署名した最初の償還パスを実行する条件 (TRUE)として機能します。

ボブがこれを償還するためには、IF句にFALSEを与えて2番目の実行経路を選択する必要がある。

<ボブのシグ> 0

Bobのロック解除スクリプトはスタックに0を置くので、IF句はBobの署名を必要とする2番目の（ELSE）スクリプトを実行させる。

二つの条件にはそれぞれ署名が必要なので、アリスは第二項を使うことができず、ボブは第一項を使うことができないのです。

条件フローは入れ子にすることができるので、ロック解除スクリプトのTRUE/FALSE値も入れ子にして、複雑な条件のパスをナビゲートすることができます。

A complex script used in Lightning Networkでは、ライトニングネットワークで使用されているような、複数の条件を持つ複雑なスクリプトの例を見ることができます。[3]ライトニング ネットワークで使用されるスクリプトは、オンチェーン フットプリントを最小化するために高度に最適化されコンパクトになっているため、読んで理解するのは簡単ではありません。それでも、本章で学んだビットコインスクリプトのコンセプトのいくつかを特定できるかどうか確認してみてください。

例1.Lightning Networkで使用されている複雑なスクリプト

DUP HASH160 <RIPEMD160(SHA256(revocationpubkey))> EQUAL

ELSE
